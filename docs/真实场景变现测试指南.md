# 真实场景变现测试指南

这份指南用于验证 Godot Agent 是否能在真实使用场景里产生可收费价值。目标不是演示“功能存在”，而是收集客户愿意付费所需的证据：省时、降风险、可交付、可复用。

## 测试结论口径

把测试结果分成三档：

| 档位 | 判定 | 适合动作 |
| :--- | :--- | :--- |
| 可售卖 | 目标用户能独立跑完核心流程，产物可直接进入项目，证据链完整 | 做付费试用、预售、外包交付 |
| 可试点 | 核心价值成立，但需要人工协助配置或解释报告 | 做 3-5 个种子用户试点 |
| 不可售卖 | 用户无法完成关键流程，或产物不能减少真实工作量 | 回到产品和交付流程修正 |

## 准备环境

在新终端中进入仓库：

```powershell
cd D:\新建文件夹\godot-agent
```

优先使用项目虚拟环境或 Python 3.12。当前系统 Python 3.14 对部分依赖不稳定，不建议作为验收环境。

```powershell
.\tools\bootstrap_clean_machine.ps1 -Preview
.\tools\bootstrap_clean_machine.ps1
python -m agent_system.cli doctor
.\tools\run_non_live_validation_shards.ps1 -Profile customer -SlowShardSeconds 60 -ContinueOnFailure
.\tools\run_release_live_gates_locally.ps1 -Preflight
```

如果要验证 Godot 编辑器联动，确认 `config.yaml` 或环境变量能找到 Godot：

```powershell
python -m agent_system.cli doctor --json
python -m agent_system.cli launch
```

变现测试必须保存这些证据：

- `logs/reports/doctor_self_check.json`
- `logs/reports/non_live_validation_shards.json`
- CLI 输出或 Portal 截图
- 生成的脚本、场景、报告、发布包路径
- 用户完成任务所用时间
- 用户是否愿意继续使用或付费

## 场景 1：独立开发者快速做可玩原型

**目标用户**：个人开发者、Game Jam 参赛者、课程学员。

**付费假设**：用户愿意为“从想法到可运行 Godot 原型”的加速付费。

**测试任务**：

```powershell
python -m agent_system.cli plan "创建一个 2D 平台跳跃原型，包括玩家移动、金币收集、敌人巡逻和血量系统"
python -m agent_system.cli run "创建一个 2D 平台跳跃原型，包括玩家移动、金币收集、敌人巡逻和血量系统" -y
python -m agent_system.cli run "审计项目资源命名" -y
```

如果 Godot 可用，继续跑：

```powershell
.\tools\run_live_sandbox_tests.ps1 -PytestArgs tests/test_live_sandbox.py -v -s
```

**验收证据**：

- `scripts/` 下出现可读、可挂载的 GDScript。
- `scenes/` 或 `sandbox_project/` 下出现可打开场景。
- 资源审计报告可解释问题，而不是只给失败。
- 用户能在 30 分钟内理解如何继续修改。

**可售卖门槛**：

- 首次使用者在 60 分钟内完成一个可运行原型。
- 生成代码需要人工重写的比例低于 30%。
- 用户愿意为单次原型生成、模板包或月订阅付费。

## 场景 2：小团队交付一个可验收切片

**目标用户**：外包团队、小型工作室、课程项目组。

**付费假设**：用户愿意为“交付前自动检查、审计、发布证据包”付费。

**测试任务**：

```powershell
python -m agent_system.cli governance --change-type feature --evidence contract,tests,docs,quality_dashboard --changed-path scenes/Main.tscn --changed-path scripts/player_controller.gd --changed-path README.md
python -m agent_system.cli production --scenario-id vertical_slice_2d --evidence contract,tests,docs,quality_dashboard --changed-path scenes/Main.tscn --changed-path scripts/player_controller.gd --changed-path README.md --mode strict
python -m agent_system.cli run "生成当前项目的发布候选检查清单" -y
```

如果要验证真实发布门禁：

```powershell
.\tools\run_non_live_validation_shards.ps1 -Profile release -SlowShardSeconds 120 -ContinueOnFailure
.\tools\run_release_live_gates_locally.ps1 -Preflight
python tools\dispatch_release_live_gates.py --project-root . --repo sossossal/godot-agent --ref main --workflow release-live-gates.yml --target-channel staging --target-environment staging --release-manifest-path api_server\static\dist\web_release_validation_ci\release_manifest.json --runner-profile-path deployment\release_live_runner_profile.json --approvers qa_lead,tech_lead,producer --providers codex,openai_api --artifact-dir logs\reports\release_live_ci --wait --wait-timeout 5400
```

**验收证据**：

- production readiness 返回 `passed`。
- release live gate 在 GitHub Actions 中返回 `success`。
- artifact bundle 能交给非开发成员复查。
- 报告能说明 blocker、warning、next action，而不是只给测试日志。

**可售卖门槛**：

- 团队愿意把它放进交付检查流程。
- 每次交付至少节省 1-2 小时人工检查。
- 报告能减少返工、漏测或沟通成本。

## 场景 3：API / MCP 集成到外部工作流

**目标用户**：工具开发者、AI Agent 平台、教育平台、内部研发平台。

**付费假设**：用户愿意为可集成的 Godot 自动化能力付费，而不是只使用本地 CLI。

**启动 API**：

```powershell
python -m api_server.main
```

**验证 HTTP API**：

```powershell
curl.exe -X POST "http://127.0.0.1:8000/execute" -H "Content-Type: application/json" -d "{\"command\":\"生成 2D 玩家移动脚本\"}"
curl.exe -X POST "http://127.0.0.1:8000/production/validate" -H "Content-Type: application/json" -d "{\"project_path\":\"D:/新建文件夹/godot-agent\",\"scenario_id\":\"vertical_slice_2d\",\"evidence\":{\"contract\":true,\"tests\":true,\"docs\":true,\"quality_dashboard\":true},\"changed_paths\":[\"scenes/Main.tscn\",\"scripts/player_controller.gd\",\"README.md\"],\"mode\":\"strict\"}"
```

**验证 Remote MCP**：

```powershell
.\tools\run_remote_mcp_live_smoke.ps1
```

**验收证据**：

- API 返回稳定 JSON，而不是只能读终端文本。
- `godot_production_validate` 和 `godot_agent_compat` 可被远端工具调用。
- 错误响应能指出缺失路径、缺失证据或环境问题。

**可售卖门槛**：

- 外部系统能在 1 天内接入至少一个自动化能力。
- 集成方不需要理解仓库内部结构也能获得结构化结果。
- 有明确计费单位：按项目、按席位、按 CI 次数或按 API 调用。

## 变现访谈问题

每个测试用户都问同一组问题，避免只收集主观好评：

1. 你原来完成这个任务要多久？
2. 这次实际用了多久？
3. 哪个输出你可以直接使用？
4. 哪个输出必须重做？
5. 如果今天收费，你愿意按次、按月还是按团队付费？
6. 你愿意付多少钱？为什么？
7. 哪个失败会让你立刻停止使用？

## 建议定价实验

从低摩擦开始，不要一开始做复杂订阅：

| 产品形态 | 建议价格 | 验证目标 |
| :--- | :--- | :--- |
| 一次原型生成服务 | 99-299 元/次 | 验证个人开发者是否愿意为省时间付费 |
| 小团队交付检查包 | 999-2999 元/项目 | 验证 QA / release evidence 是否有交付价值 |
| 私有化集成试点 | 3000-10000 元/试点 | 验证 API/MCP 是否能进入客户工具链 |
| 月订阅 | 49-199 元/席/月 | 仅在重复使用成立后测试 |

## 停止条件

出现以下情况，不要继续推销售卖，先修产品：

- 新用户无法完成 bootstrap 或 doctor。
- 生成物无法在 Godot 中打开。
- release / production 报告不能解释失败原因。
- 用户只觉得“有趣”，但说不出节省了什么成本。
- 三个测试用户里没有一个愿意给出明确付费方式。

## 当前推荐测试顺序

1. 先跑独立开发者原型，验证核心生成价值。
2. 再跑小团队交付切片，验证质量和发布证据价值。
3. 最后跑 API/MCP 集成，验证能否进入外部工作流。

如果三类场景里至少两类达到“可售卖”，就可以开始做付费试点；如果只有一类成立，优先把产品定位收窄到那一类用户。
