# Godot Agent 使用指南

## 三种入口

### 1. CLI

推荐优先使用 CLI，因为它和当前实现最一致。

```bash
python -m agent_system.cli doctor
python -m agent_system.cli plan "创建一个玩家场景并生成移动脚本"
python -m agent_system.cli run "生成 2D 玩家移动脚本" -y
python -m agent_system.cli history --limit 5
python -m agent_system.cli roles
python -m agent_system.cli chat
```

### 2. Python API

`GodotAgentRouter.execute()` 返回 `Task`，需要从 `status`、`logs`、`artifacts`、`context` 读取结果。

```python
from agent_system.router import GodotAgentRouter
from agent_system.models import TaskStatus

agent = GodotAgentRouter()
task = agent.execute("生成 2D 玩家移动脚本")

if task.status == TaskStatus.SUCCESS:
    print(task.message)

    script = next((a for a in task.artifacts if a.type == "script"), None)
    if script:
        print(script.path)
        print(script.content)
else:
    print(task.status.value)
    print(task.message)
```

### 3. HTTP API

```bash
python api_server/main.py
```

```python
import os
import requests

api_host = os.environ.get("GODOT_AGENT_API_HOST", "127.0.0.1").strip() or "127.0.0.1"
api_port = int(os.environ.get("GODOT_AGENT_API_PORT", "8000"))

response = requests.post(
    f"http://{api_host}:{api_port}/execute",
    json={"command": "生成玩家移动脚本"}
)

task = response.json()
print(task["status"])
print(task["message"])

script = next((a for a in task["artifacts"] if a["type"] == "script"), None)
if script:
    print(script["path"])
```

如需改端口，可先设置:

```bash
set GODOT_AGENT_API_HOST=127.0.0.1
set GODOT_AGENT_API_PORT=8011
```

计划编辑相关接口:

```bash
curl -X POST "http://127.0.0.1:8011/plan" \
  -H "Content-Type: application/json" \
  -d "{\"command\": \"创建一个玩家场景并生成移动脚本\"}"
```

```bash
curl -X POST "http://127.0.0.1:8011/execute-plan" \
  -H "Content-Type: application/json" \
  -d "{\"project_path\": \"default\", \"prompt\": \"创建一个玩家场景并生成移动脚本\", \"steps\": [{\"name\": \"Logic\", \"description\": \"处理逻辑\", \"role\": \"code_generator\"}]}"
```

如果未设置环境变量，把上面的端口换回默认的 `8000` 即可。

## 结果结构

当前版本最常用的字段:

- `task.status`: `pending` / `running` / `success` / `failed` / `rolled_back`
- `task.message`: 适合直接展示给用户的摘要
- `task.steps`: 规划出的多步任务
- `task.logs`: 执行过程日志
- `task.artifacts`: 生成物，例如 `scene`、`script`、`editor_script`
- `task.context`: 执行上下文，例如 `release_url`

## 常见示例

### 场景与脚本

```python
task = agent.execute("创建一个名为 MainScene 的 2D 场景并生成移动脚本")
print(task.status.value)
for step in task.steps:
    print(step.role, step.name, step.status.value)
```

### 读取生成脚本

```python
task = agent.execute("生成血量系统代码")
script = next((a for a in task.artifacts if a.type == "script"), None)

if script:
    with open("health_system.gd", "w", encoding="utf-8") as f:
        f.write(script.content)
```

### 查看角色能力

```python
for role_name in agent.get_available_roles():
    info = agent.get_role_info(role_name)
    print(role_name, info["description"])
```

### 查看历史

```python
history = agent.get_history(limit=5)
for item in history:
    print(item["prompt"], item["status"], item.get("role"))
```

## 资源审计输出

`审计项目资源命名` 会生成 `logs/reports/resource_audit_<timestamp>.md` 报告，并在 `task.context` 中写入:

- `audit_issue_count`
- `audit_file_issue_count`
- `audit_scene_issue_count`
- `audit_resource_issue_count`
- `audit_import_issue_count`
- `audit_error_count`
- `audit_warning_count`
- `audit_info_count`
- `audit_highest_severity`
- `audit_scene_header_issue_count`
- `audit_scene_reference_issue_count`
- `audit_resource_header_issue_count`
- `audit_resource_reference_issue_count`
- `audit_binary_resource_issue_count`
- `audit_cycle_issue_count`
- `audit_import_config_issue_count`
- `audit_import_artifact_issue_count`

自动修复相关命令:

- `预览修复项目资源命名`: 只生成 `logs/reports/resource_fix_preview_<timestamp>.md`
- `修复项目资源命名`: 应用低风险修复，并生成 `logs/reports/resource_fix_apply_<timestamp>.md`

自动修复目前覆盖:

- 文件重命名
- `.import` sidecar 文件同步重命名
- `.import` 中 `source_file` 的逻辑路径修正
- `.gd` / `.tscn` / `.tres` / `.res` / `.import` 中 `res://` 文本引用更新
- 可安全推断的 `.godot/imported` 产物重命名

自动修复目前不会直接处理:

- 目录重命名
- 场景节点重命名
- UID 缺失或冲突
- 引用环断开
- 结构损坏类问题，例如缺失 `ext_resource id`

当前审计范围包括:

- 文件和目录命名
- `.tscn` 中 `uid` / `load_steps` / `format` 等头部字段有效性
- 跨场景重复 `uid`
- `.tres` / `.res` 中 `type` / `format` / `uid` 等头部字段有效性
- 跨场景和跨资源文件的重复 `uid`
- `.tscn` 场景节点命名
- `.tscn` 中 `ext_resource` / `sub_resource` 声明、目标路径和引用有效性
- `.tscn` 中 `ext_resource uid` 与目标场景 `uid` 的交叉一致性
- `.tscn` 中 `ext_resource uid` 与目标 `.tres` / `.res` 的交叉一致性
- `.tscn` 中重复声明的 `ext_resource path`
- `.tres` / `.res` 中 `ext_resource` / `sub_resource` 声明、目标路径和引用有效性
- `.tres` / `.res` 中 `ext_resource uid` 与目标资源 `uid` 的交叉一致性
- `.tres` / `.res` 中重复声明的 `ext_resource path`
- `.tscn` / `.tres` / `.res` 之间的引用环，报告会给出最短闭环路径和精确到 `文件:行号` 的断环建议
- 二进制 `.res` 的降级识别和跳过深度解析提示
- `.import` 资源文件命名
- `.import` 中 `source_file` 指向的源资源命名
- `.import` 中 `importer` / `type` / `source_file` 关键字段一致性
- `.import` 中 `dest_files` 指向的 `.godot/imported` 产物存在性

报告格式补充:

- 报告摘要会列出 `Errors`、`Warnings`、`Infos`
- 每条问题会带 `[ERROR]`、`[WARNING]` 或 `[INFO]` 标签
- Portal 审计面板支持直接打开报告文件
- Portal 审计面板支持按条目查看对应源码片段和定位行，源码预览头部会直接显示脚本 `class_name / func / signal` 或场景节点上下文，并提供可点击 badge 重新精确打开对应类、函数、信号、节点或实例源
- Portal 审计面板支持把条目直接发送到 Godot 编辑器里打开，脚本会尽量带行号定位，并额外返回最近的 `class_name / func / signal` 上下文；`.tscn` 会按定位行尽量选中对应节点，并在同名节点场景下优先显示完整节点路径；如果命中的是实例化分支，还会带实例根路径和实例源场景；`.import` 会优先跳到源资源，场景和资源其余情况走最佳努力打开
- Portal 项目状态区会显示最近一次编辑器回执，用来确认资源是否真的已经在 Godot 端打开
- Portal 产物中心会从最近任务历史中汇总脚本、报告、场景、截图、发布产物，支持按类型筛选、内联预览、直接打开文件，以及把 `res://` 产物再次发送到 Godot 编辑器中打开

## 安全重构

当前支持的自然语言重构入口:

- `重命名类 PlayerController 为 HeroController`
- `重命名函数 move_player 为 move_character`
- `重命名信号 jumped 为 landed`

行为说明:

- 会自动备份被修改文件
- 当前会扫描 `.gd`、`.tscn`、`.tres`、`.res` 和 `project.godot`
- 类重命名会更新 `class_name` 和常见文本引用
- 函数重命名会更新定义、直接调用、`call/has_method/Callable` 等常见引用
- 信号重命名会更新定义、`.connect/.emit`、`emit_signal/connect` 等常见引用

## 端到端测试

除了 `运行场景测试`，现在还支持:

- `端到端测试 res://scenes/main_scene.tscn`
- `端到端测试 res://scenes/main_scene.tscn 并向右跳跃后截图`

行为说明:

- tester 会生成 headless 冒烟脚本并通过 `GodotCLI.run_headless()` 执行
- 当前内置了简单输入回放关键词，例如 `向右`、`向左`、`跳`、`交互`
- 包含 `截图/快照` 时会声明截图产物路径，真实保存仍取决于 Godot 运行环境

## 计划编辑器

Portal 里现在支持先规划、后编辑、再执行:

- 输入指令后点击“规划”
- 在右侧计划编辑器里修改步骤名、说明、角色
- 也可以删除步骤、调整上下顺序、手动添加新步骤
- 点击“执行当前计划”后会调用 `/execute-plan`

当前编辑器默认覆盖这几类操作:

- 删改步骤
- 改角色
- 改顺序
- 添加额外步骤

当前还没有做更细的依赖图编辑，`depends_on` 仍以接口字段为主，前端暂未单独暴露。

## 编辑器上下文同步

当前 Godot 插件会持续上报这些状态:

- 当前场景路径
- 选中节点名称
- 选中节点完整路径
- 选中节点详情，例如节点类型、绑定脚本、owner 路径
- 当前打开脚本路径
- 当前脚本光标行列
- Inspector 当前资源路径 / 类型，或当前 Inspector 节点路径

服务端会把这些字段放进 `/health` 的 `editor_state`，并在任务执行时自动注入到 `task.context["editor_state"]`。对于当前脚本，服务端还会按光标行补全最近的 `class_name / func / signal` 上下文。

这带来的直接效果是:

- Portal 项目状态区会显示更完整的实时上下文
- `developer` 角色在“当前/选中节点下添加节点”时，会优先走完整节点路径，减少同名节点歧义
- `tester` 在没有显式 `res://` 场景路径时，会优先使用当前打开场景
- `code_generator` 在“重命名当前类 / 当前函数 / 当前信号 ...”这类指令下，会直接用当前脚本和光标位置推断目标符号，例如 `重命名当前函数为 move_character`

## 环境要求

- 场景创建、场景测试、Web 导出需要 Godot 可执行文件。
- 实时节点注入、属性修改、截图同步需要 Godot 编辑器插件联动。
- 没有 Godot 环境时，相关任务可能失败并回滚，这是预期行为。

## 故障排查

### 1. `doctor` 先跑一遍

```bash
python -m agent_system.cli doctor
```

### 1.1 启动编辑器

```bash
python -m agent_system.cli launch
python -m agent_system.cli launch --scene res://scenes/Main.tscn
```

如果你走的是 API / Portal 链路，也可以直接请求 `POST /editor/launch`。另外 `/execute`、`/execute-plan` 和 `/editor/open-resource` 支持传 `auto_launch_editor=true`，编辑器离线时会先拉起 Godot，再等待插件上线继续执行。

如果你要跑仓库自带的 `sandbox_project/` live 联调，Windows PowerShell 下可以直接使用：

```powershell
.\tools\run_live_sandbox_tests.ps1 -ApiPort 8011
```

需要分步排查时，再改用 `start_live_sandbox.ps1` 和 `stop_live_sandbox.ps1` 手动控制启动与清理。

### 2. 检查 `config.yaml`

```yaml
godot:
  executable_path: "C:/Godot/Godot_v4.x_stable_win64.exe"
  project_path: "D:/MyGodotProject"
```

`executable_path` 也可以留空。系统会继续按下面顺序查找：

- 环境变量 `GODOT` / `GODOT_EXE` / `GODOT_PATH`
- 系统 `PATH` 中的 `godot` / `godot4`

其中环境变量既可以写完整可执行文件路径，也可以写 Godot 所在目录。

`doctor` 现在会把命中来源直接打印出来，例如 `config.yaml`、`环境变量 GODOT` 或 `PATH godot4`；Portal 项目状态区也会显示当前 Godot 来源，并可展开查看、复制完整可执行文件路径。

### 3. 场景/测试任务失败

先确认:

- Godot 可执行文件能被找到
- `project_path` 指向真实项目
- 需要编辑器联动的任务是否已启用插件
