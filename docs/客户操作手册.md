# Godot Agent 客户操作手册

本手册面向试用客户、交付团队和内部评估人员。目标是在一台 Windows 机器上完成安装、自检、真实任务执行、结果验收和问题反馈。

## 适用对象

| 角色 | 你可以完成什么 |
| :--- | :--- |
| 独立开发者 | 快速生成脚本、原型场景、基础玩法系统 |
| 小型团队 | 做交付前审计、生产验证、发布候选检查 |
| 技术负责人 | 验证 API、MCP、CI gate 和交付证据是否能接入现有流程 |
| 试用客户 | 判断工具是否节省时间、降低返工、形成可交付证据 |

## 使用前准备

### 必需环境

- Windows 10/11
- Python 3.12
- Git
- Godot 4.x，用于真实场景、编辑器联动和导出验证
- Chrome 或 Edge，用于 Portal 浏览器验证

### 可选环境

- GitHub CLI，用于触发远端 release live gate
- GitHub token，用于真实 GitHub Actions workflow dispatch
- 固定 Windows self-hosted runner，用于完整发布级验证

## 第一次安装

打开 PowerShell：

```powershell
cd D:\新建文件夹\godot-agent
.\tools\bootstrap_clean_machine.ps1 -Preview
.\tools\bootstrap_clean_machine.ps1
```

安装完成后执行自检：

```powershell
python -m agent_system.cli doctor
```

期望结果：

- 终端显示自检通过或明确修复动作。
- 生成 `logs/reports/doctor_self_check.json`。
- 如果 Godot 没有配置，报告会指出需要设置 Godot 路径。

建议在首次试用前再跑一次面向客户演练的非 live 分片和 release-live 本地预检：

```powershell
.\tools\run_non_live_validation_shards.ps1 -Profile customer -SlowShardSeconds 60 -ContinueOnFailure
.\tools\run_release_live_gates_locally.ps1 -Preflight
.\tools\run_pr_release_gate.ps1 -Stage customer -Mode preflight
.\tools\run_pr_release_gate.ps1 -Stage customer -ContinueOnFailure
.\tools\export_customer_trial_bundle.ps1 -GateMode preflight -ContinueOnFailure
.\tools\export_customer_trial_bundle.ps1 -GateMode preflight -SyncPluginBeforeDoctor -PrepareReleaseFixture -RestorePreparedFixture -ContinueOnFailure
```

第一条会覆盖 API、P19 创建流程、资源质量、Agent/MCP 和 release foundation；第二条只检查 release-live gate 需要的 Python、PowerShell、manifest、runner profile、config、browser path 和 runner labels，不会执行真实 live 步骤；第三条是客户试用轻量预检；第四条会把客户试用 gate 总结到 `logs/reports/pr_release_gate/gate_summary.json` 和 Markdown；第五条会把 doctor、customer gate、相关报告、`recommended_actions`、`customer_trial_readiness.json`、`customer_trial_commands.json` 和 `rerun_customer_trial.ps1` 汇总到 `logs/reports/customer_trial_bundle/`，其中 readiness JSON 和 manifest 会记录 `release_manifest_path`，bundle Markdown 顶部也会显示本轮 release manifest，readiness JSON 还会内嵌 `evidence_files`、`evidence_file_count` 和 `missing_evidence_files`，bundle Markdown 也会显示证据数量和缺失状态，方便只看 readiness 或 Markdown 时直接定位 gate/fixture/doctor 证据并确认清单没有断链；第六条在 preflight 下只生成 synthetic release manifest、notes 和 build log，适合没有现成 release 产物的试用机器；完整 runtime reports 只在 full gate 或正式 release workflow 中生成。
追加 `-SyncPluginBeforeDoctor` 时，会在 doctor 前以 `addons/godot_agent` 为单一来源同步 `godot_plugin` 和 `sandbox_project` 的插件副本，减少试用机因插件副本漂移产生的假阻断。
追加 `-RestorePreparedFixture` 时，演练结束会恢复 synthetic fixture 改写过的受管 release/runtime 文件，只保留本轮 gate 和客户证据包输出。
追加 `-ContinueOnFailure` 时，bundle 会把 `continue_on_failure` 写入 preview、readiness、manifest 和 Markdown，便于区分“遇阻停止”和“遇阻继续收集证据”的演练结果。
`customer_trial_readiness.json` 中的 `readiness_level` 会区分 `ready / needs_attention / blocked`，其中 `needs_attention` 表示 gate 已通过但仍有建议动作需要试用负责人确认。
`recommended_action_items` 会保留建议动作来源和检查 ID，方便审查工具区分 doctor 修复项和 release-live preflight 提醒；`recommended_actions` 字符串列表仍保留给人工阅读。preview 会显示 `status=preview`、`readiness_level=preview`、计划 `planned_step_count` / `planned_step_ids`、`skipped_step_count=0` / `skipped_step_ids=[]`、空 `blocked_steps` / `results` / `recommended_actions` / `recommended_action_items` / `evidence_files` / `missing_evidence_files`、零值 `passed_count` / `blocked_count` / `status_counts` / `total_duration_seconds` / `slowest_step_seconds` / `recommended_action_count` / `evidence_file_count`、空 `slowest_step_id`、兼容字段 `step_count` / `step_ids`、`command_count` 与 `command_ids`，readiness、bundle manifest 和 Markdown 摘要会同时显示 `total_duration_seconds`、`slowest_step_id`、`slowest_step_seconds`、`planned_step_count`、`planned_step_ids`、`skipped_step_count`、`skipped_step_ids`、实际执行的 `step_count` / `step_ids`、`passed_count`、`blocked_count`、`recommended_action_count`、`command_count` 与 `command_ids`，Markdown 顶部也会直接列出计划、跳过和实际执行的步骤 ID，并提供 `status_counts` 给工具读取，方便快速判断耗时、最慢步骤、计划执行面、未执行步骤、实际执行面、通过/阻断规模、修复项和复跑命令规模。
正式发布前或付费试用前可以追加 `-FailOnNeedsAttention`，把 `needs_attention` 升级为非零退出码；普通探索性试用不建议开启。bundle Markdown 摘要会显示 `browser_path`、`fail_on_needs_attention` 和实际 `should_fail_on_needs_attention`。
如果 release-live preflight 需要浏览器 lane，把 Chrome 或 Edge 可执行文件路径传给 `-BrowserPath`，例如 `-BrowserPath "C:\Program Files\Google\Chrome\Application\chrome.exe"`。

## 配置 Godot

系统按以下顺序查找 Godot：

1. `config.yaml` 中的 `godot.executable_path`
2. 环境变量 `GODOT` / `GODOT_EXE` / `GODOT_PATH`
3. 系统 `PATH` 中的 `godot` / `godot4`

临时设置示例：

```powershell
$env:GODOT_EXE = "D:\迅雷下载\Godot\godot.EXE"
python -m agent_system.cli doctor
```

启动编辑器：

```powershell
python -m agent_system.cli launch
```

## 快速试用流程

### 1. 查看任务计划

```powershell
python -m agent_system.cli plan "创建一个 2D 玩家移动脚本，并说明如何挂到角色节点"
```

你应该看到：

- 任务拆解步骤
- 每一步角色分工
- 是否需要 Godot 或文件写入

### 2. 执行脚本生成

```powershell
python -m agent_system.cli run "生成 2D 玩家移动脚本" -y
```

验收方式：

- 查看终端任务状态是否为成功。
- 查看 `scripts/` 下是否生成脚本。
- 打开脚本，确认变量、输入、移动逻辑可读。

### 3. 执行资源审计

```powershell
python -m agent_system.cli run "审计项目资源命名" -y
```

验收方式：

- 查看 `logs/reports/` 下的资源审计报告。
- 确认报告能说明 error / warning / info。
- 如果存在低风险修复需求，先使用预览：

```powershell
python -m agent_system.cli run "预览修复项目资源命名" -y
```

## 真实 Godot 联动验证

如果你要验证编辑器联动、节点注入、场景创建、截图和 WebSocket 事件：

```powershell
.\tools\run_live_sandbox_tests.ps1 -PytestArgs tests/test_live_sandbox.py -v -s
```

期望结果：

- live sandbox 自动启动 API 和 Godot。
- 测试结束后自动清理进程。
- 所有 live sandbox 测试通过。

常见失败原因：

| 现象 | 处理 |
| :--- | :--- |
| 找不到 Godot | 设置 `GODOT_EXE` 或修改 `config.yaml` |
| API 端口被占用 | 设置 `GODOT_AGENT_API_PORT` 到空闲端口 |
| 编辑器未上线 | 确认 Godot 可执行文件能手动打开项目 |
| 权限或 ACL 报错 | 运行 `.\tools\fix_acl_artifacts.ps1 -Preview` 查看修复建议 |

## Portal 使用

启动 API：

```powershell
python -m api_server.main
```

打开：

```text
http://127.0.0.1:8000/portal/index.html
```

Portal 可用于：

- 查看任务历史
- 执行计划和复审
- 查看生产验证
- 查看 Release Candidate
- 查看 Build / Run Matrix
- 查看 Release Promotion / Execution / Live CI
- 打开产物中心

如果端口不是 `8000`，把 URL 中端口换成实际端口。

## API 集成

### 执行自然语言任务

```powershell
curl.exe -X POST "http://127.0.0.1:8000/execute" `
  -H "Content-Type: application/json" `
  -d "{\"command\":\"生成 2D 玩家移动脚本\"}"
```

### 生产验证

```powershell
curl.exe -X POST "http://127.0.0.1:8000/production/validate" `
  -H "Content-Type: application/json" `
  -d "{\"project_path\":\"D:/新建文件夹/godot-agent\",\"scenario_id\":\"vertical_slice_2d\",\"evidence\":{\"contract\":true,\"tests\":true,\"docs\":true,\"quality_dashboard\":true},\"changed_paths\":[\"scenes/Main.tscn\",\"scripts/player_controller.gd\",\"README.md\"],\"mode\":\"strict\"}"
```

通过标准：

- `readiness_status` 为 `passed`。
- `blocking_checks` 为空。
- 返回结果包含各阶段检查详情。

## 交付级验证

### 本地 production gate

```powershell
python -m agent_system.cli production `
  --scenario-id vertical_slice_2d `
  --evidence contract,tests,docs,quality_dashboard `
  --changed-path scenes/Main.tscn `
  --changed-path scripts/player_controller.gd `
  --changed-path README.md `
  --mode strict
```

### 真实 release live gate

前提：

- 已安装 GitHub CLI。
- 已配置 `GH_TOKEN` 或 `GITHUB_TOKEN`。
- self-hosted Windows runner 在线。

```powershell
python tools\dispatch_release_live_gates.py `
  --project-root . `
  --repo sossossal/godot-agent `
  --ref main `
  --workflow release-live-gates.yml `
  --target-channel staging `
  --target-environment staging `
  --release-manifest-path api_server\static\dist\web_release_validation_ci\release_manifest.json `
  --runner-profile-path deployment\release_live_runner_profile.json `
  --approvers qa_lead,tech_lead,producer `
  --providers codex,openai_api `
  --artifact-dir logs\reports\release_live_ci `
  --wait `
  --wait-timeout 5400
```

通过标准：

- GitHub Actions run 结论为 `success`。
- `Run full live validation` 成功。
- `Export live release CI artifacts` 成功。
- `Upload live release artifacts` 成功。

## 典型客户任务


- 没有 blocker。

## 已知限制

- 当前公开生产发布还需要客户自己的签名证书、外部分发渠道和发布账号。
- 当前已有 request auth、identity registry 和 handoff，但不是完整企业身份系统。
- release live gate 依赖固定 self-hosted Windows runner。
- 没有 Godot 环境时，场景创建、截图、导出和编辑器联动无法完整验证。
- 商业变现判断必须结合真实客户试用反馈，不能只看自动化测试结果。

## 推荐试用顺序

1. 先跑 `doctor` 和基础脚本生成。
2. 再跑资源审计和 production validate。
3. 有 Godot 环境时跑 live sandbox。
4. 团队客户再跑 release live gate。
5. 技术集成客户最后验证 HTTP API 和 Remote MCP。
