# 游戏开发标准化增强路线图

这份文档聚焦“如何让后续增强更稳定”，不是单纯罗列新功能。

目标有 4 个：

- 约束所有 AI 和工具输出，减少后期返工
- 约束文件树和产物落点，降低项目越做越乱的风险
- 按游戏类型沉淀通用模板，提高复用率
- 延续 `skills` 化思路，让不同 AI API 接入后仍保持质量、结构和测试一致

在这 4 个方向之外，还建议把以下底座一起纳入路线图：

- 统一契约层
- 命名与目录规范检查器
- 模板注册中心
- Skill 质量基线
- 可回放测试资产
- 预算系统
- 运营事件字典
- 迁移与兼容层

## 总体原则

### 1. 输出先有契约，再有实现

所有模块都应先定义结构化输入输出，再写逻辑。至少覆盖：

- `Task.context`
- `artifacts`
- `history`
- `release metadata`
- `data table schema`
- `telemetry events`
- `performance reports`

每个结构都应带：

- `schema_version`
- 必填字段
- 默认值
- 枚举范围
- 失败码或错误类别

### 2. 文件树先有规则，再允许生成

任何新资源、脚本、报告、导出产物都不应直接自由落盘。应先定义：

- 允许写入的目录白名单
- 文件命名规则
- 同类资源的固定落点
- 自动创建目录时的父级约束
- 重复命名和孤儿文件检查

### 3. 模板先注册，再允许实例化

模板不应散落在各 skill 内部匿名存在。需要一个统一注册中心，至少记录：

- 模板名称
- 模板版本
- 适用游戏类型
- 可覆盖参数
- 输出目录策略
- 依赖的资源和测试模板

### 4. Skill 先满足质量壳，再允许接 AI

不同 AI API 只负责“生成内容”，不能直接决定项目结构。每个 skill 都应自带统一壳层：

- 输入校验
- 目录校验
- 模板加载
- 生成后验证
- 产物摘要
- 回滚锚点
- 目标测试

## P0: 必须先补的标准化底座

### 1. 统一契约层

目标：

- 让内容生成、发布、遥测、性能、数据表都共享同一套结构化约束

建议新增：

- `agent_system/contracts/` 目录
- 统一的 schema 定义与校验入口
- `task / artifact / telemetry / performance / template manifest` 契约文件

完成标准：

- 新模块接入前必须先定义契约
- Portal、CLI、Router、MCP 输出字段一致
- 历史记录和发布产物可稳定反序列化

### 2. 文件树与命名规范引擎

目标：

- 把“约定目录”升级成“自动阻断错误落盘”

建议覆盖：

- 脚本目录
- 场景目录
- 资源目录
- 数据表目录
- 报告目录
- 发布目录
- 遥测和性能基线目录

建议规则：

- 禁止直接写入仓库根目录
- 禁止把测试基线混进运行目录
- 限制文件后缀与目录组合
- 检查重复资源、孤儿资源、错误命名

完成标准：

- 所有创建类 skill 在落盘前都会通过目录检查
- `doctor` 或独立命令可审计全项目目录健康度

### 3. 模板注册中心

目标：

- 把“按游戏类型建立通用模板”从文档建议变成真实机制

首批模板建议按类型分层：

- `platformer`
- `topdown_action`
- `arpg`
- `roguelike`
- `tower_defense`
- `visual_novel`
- `survival_crafting`

每个类型至少包含：

- 目录骨架
- 核心场景模板
- 基础脚本模板
- 默认数据表
- UI 模板
- 测试模板
- 性能预算模板

完成标准：

- 新项目初始化时可选择类型模板
- 模板升级有版本号和迁移说明

### 4. Skill 统一基线

目标：

- 让后续所有增强模块沿用相同交付模式

每个 skill 至少应输出：

- `summary`
- `artifacts`
- `validation`
- `rollback`
- `schema_version`
- `quality_gate`

每个 skill 至少应包含：

- 输入参数 schema
- 输出 schema
- 文件落点规则
- 模板依赖说明
- 最小回归测试

完成标准：

- 新增 skill 不再是“单个脚本功能”，而是完整的可验证模块

### 5. 预算系统与回放资产

目标：

- 给 QA、性能、画面差异建立统一基线

首批基线资产：

- golden screenshots
- 输入回放脚本
- 标准测试场景
- 标准数据表样本
- 性能基准结果

首批预算指标：

- `scene_load_ms`
- `fps_min`
- `memory_peak_mb`
- `draw_call_budget`
- `node_count_budget`
- `texture_budget_mb`

完成标准：

- 发布门禁不再只依赖单次运行结果，而有稳定对照基线
