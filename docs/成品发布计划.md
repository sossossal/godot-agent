# 成品发布计划

面向“把当前仓库从技术可用工具，推进到可交付成品”的执行版计划。

这份文档不是功能愿望清单，而是按真实发布阻塞项排序的落地路线。

## 目标

把当前 Godot 多角色 Agent 系统推进到下面这类可交付状态：

- 新机器能按文档完成安装、启动、联调和基础验证
- 单个功能可以从计划、执行、评审、验收到归档完整追踪
- `qa / staging / release` 渠道的出包门禁默认 fail closed
- 发布包、回滚锚点、验证证据、执行记录可稳定追溯
- 支持至少一条清晰的对外交付形态：
  - 本地开发者版
  - 团队内测版
  - 受控生产版

## 当前判断

仓库已经具备较完整的执行骨架：

- `Router / Roles / CLI / API / Portal / Plugin` 主链可用
- `release_candidate / release_promotion / release_execution` 已经落地
- `quality / governance / production / agent compatibility` 已有可执行检查
- live / browser / remote MCP 已有自动化验证入口

当前更大的风险不再是“功能做不出来”，而是：

- 验收是否足够硬
- 发布是否足够稳
- 安装与支持是否足够清晰
- 安全与权限边界是否足够收紧
- 团队是否能在同一条任务链上协作

## 发布原则

发布成品时，优先遵守这五条：

1. 默认阻断，不默认放行。
2. 结构化记录优先于聊天记录和口头约定。
3. clean machine 可复现优先于“作者机器可用”。
4. 渠道、版本、证据、回滚必须一一对应。
5. 先收紧主链路，再扩新能力。

## 不建议现在优先做的事

下面这些可以做，但不该排在主链之前：

- 新增更多零散生成 skill
- 更复杂的风格化 UI
- 更多模板市场能力
- 更重的多 provider 扩展

原因很简单：这些不会先解决“怎么稳定出包、怎么明确验收、怎么支持用户”。
