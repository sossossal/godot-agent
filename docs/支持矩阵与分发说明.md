# 支持矩阵与分发说明

这份文档收口当前仓库对外可承诺的安装路径、支持边界和分发形态。

重点不是“理论上也许能跑”，而是“当前仓库已经有脚本、测试或 contract 能支撑什么”。

## 当前推荐分发形态

### 1. 本地开发者版

适用对象：

- 直接 clone 仓库并在本机执行 CLI / API / Portal / Godot 联调的开发者

交付内容：

- 整个仓库
- `config.yaml`
- `.venv` 或系统 Python
- `addons/godot_agent/` 作为插件单一源码
- `godot_plugin/` 与 `sandbox_project/` 作为同步副本

推荐入口：

```powershell
.\tools\bootstrap_clean_machine.ps1 -Preview
.\tools\bootstrap_clean_machine.ps1
```

需要补一次 targeted non-live 回归时：

```powershell
.\tools\bootstrap_clean_machine.ps1 -IncludeSmoke
```

### 2. 团队内测 / 发布执行版

适用对象：

- 需要消费 `release_candidate / release_promotion / release_execution` 主链的 QA、制作和发布执行者

交付内容：

- 本地开发者版全部内容
- `release_manifest.json / release_notes.md / qa_gate_report.md`
- `deployment/release_promotion_history.json`
- `deployment/release_live_runner_profile.json`
- Portal 发布面板与 Review / Evidence / Rehearsal 报告导出

推荐补充验证：

```powershell
.\tools\run_p5_production_checks.ps1
.\tools\run_full_live_validation.ps1
```

### 3. 版本化分发 Bundle

适用对象：

- 需要把当前 release 目录和验证证据交给 QA、制作、发布执行或受控内测环境，但还没切到真正安装器/外部分发渠道的团队

交付内容：

- `logs/reports/release_distribution/<channel>/<build_id>/distribution_manifest.json`
- `install_release_bundle.ps1 / upgrade_release_bundle.ps1 / uninstall_release_bundle.ps1`
- `support_matrix.md / release_manifest.json / release_notes.md / qa_gate_report.md`
- `installed_release.example.json`
- `release_payload/` 下的版本化发布内容

推荐入口：

```powershell
python .\tools\export_release_distribution_bundle.py `
  --project-root . `
  --runtime-root . `
  --channel staging `
  --target-environment staging
```

说明：

- 汇总状态会写到 `logs/reports/release_distribution_bundle_<channel>.json`
- `Release Promotion / Review Bundle / Deployment Rehearsal / Release Execution Report` 会直接复用这份 bundle 状态
- 如果你要把安装链也沉淀成证据，可继续执行：

```powershell
python .\tools\export_release_distribution_install_smoke.py `
  --project-root . `
  --runtime-root . `
  --channel staging `
  --target-environment staging
```

- 这条命令会执行一次临时 `install -> upgrade -> uninstall`，把结果写到 `logs/reports/release_distribution_install_smoke_<channel>.json`
- smoke 报告现在会额外沉淀 `state_path / installed_build_id / installed_version / previous_build_id / backup_dir / removed_build_id / removed_version`，并同步回写到 `release_distribution_bundle_<channel>.json`
- `release` 目标下如果 distribution bundle 没通过 install smoke，promotion checklist 和 rehearsal 仍会直接阻断
- 如果你还要把这份已验证 bundle 输出成对外交付包，可继续执行：

```powershell
python .\tools\export_release_distribution_archive.py `
  --project-root . `
  --runtime-root . `
  --channel staging `
  --target-environment staging
```

- 这条命令会生成 `logs/reports/release_distribution_packages/<channel>/<build_id>/release_distribution_bundle.zip` 与 `release_distribution_bundle.sha256`
- `release_distribution_bundle_<channel>.json` 会同步刷新 archive 状态；`release` 目标下如果 archive 缺失，promotion checklist 和 rehearsal 仍会直接阻断
- 如果你还要补齐渠道 latest / releases 索引，可继续执行：

```powershell
python .\tools\export_release_distribution_channel_index.py `
  --project-root . `
  --runtime-root . `
  --channel staging `
  --target-environment staging
```

- 这条命令会生成 `logs/reports/release_distribution_channels/<channel>/latest.json`、`releases.json` 与 `logs/reports/release_distribution_channel_<channel>.json`
- `release_distribution_bundle_<channel>.json` 会同步刷新 channel index 状态；`release` 目标下如果渠道索引缺失或没有指向当前 archive，promotion checklist 和 rehearsal 仍会直接阻断
- 如果你还要把这份 verified archive + 渠道索引继续收成一个可直接 handoff 的安装包目录，可继续执行：

```powershell
python .\tools\export_release_distribution_handoff.py `
  --project-root . `
  --runtime-root . `
  --channel staging `
  --target-environment staging
```

- 这条命令会在 `logs/reports/release_distribution_handoff/<channel>/<build_id>/` 下生成 `distribution_handoff_manifest.json`、`install_release_handoff.ps1`、`upgrade_release_handoff.ps1`、`uninstall_release_handoff.ps1`、`packages/release_distribution_bundle.zip(.sha256)`、`channel/latest.json`、`channel/releases.json` 与顶层 `release_manifest / release_notes / qa_gate_report / support_matrix` 副本
- wrapper 脚本会临时解压 bundle archive，再转调 bundle 内部的 install/upgrade/uninstall 脚本，所以消费侧不必继续维护原仓库里的 `logs/reports/release_distribution_*` 路径
- `release_distribution_bundle_<channel>.json` 会同步暴露 `handoff_status / handoff_dir / handoff_manifest_path`，方便 promotion / execution / CI artifact 直接判断 portable handoff 包是否存在且完整
- 如果还要把 verified archive 收成可交给外部签名环节消费的 intake 包，可继续运行 `python .\tools\export_release_distribution_signing_handoff.py --project-root . --runtime-root . --channel release --target-environment production`；它会生成 `logs/reports/release_distribution_signing/<channel>/<build_id>/distribution_signing_manifest.json`、`SIGNING_INSTRUCTIONS.md`、`unsigned/release_distribution_bundle.zip(.sha256)` 与 `metadata/` 下的 release manifest / notes / QA gate / support matrix / channel index 副本，并把 `signing_handoff_*` 状态回写到 `release_distribution_bundle_<channel>.json`
- 这已经补齐“版本化分发目录 + install/upgrade/uninstall 实际 smoke + 可交付 archive + 渠道 latest/releases 索引”，但还不是完整的外部安装器、签名包或渠道分发体系
- `.github/workflows/release-validation.yml` 的 non-live artifact 也会额外带上 `release_distribution_handoff/`，让 staging rehearsal 证据直接包含可 handoff 的安装包目录
- `.github/workflows/release-live-gates.yml` 会把这条 bundle/archive/channel index/handoff/signing handoff 链路和真实 `run_full_live_validation.ps1` 串成 release 级 CI 入口，并通过 `tools/export_release_live_ci_artifacts.py` 把 runtime evidence 收成独立 artifact bundle
- `deployment/release_live_runner_profile.json` 现在会作为 self-hosted runner 的受管 manifest 被 baseline 和 CI artifact 直接消费；它至少能声明 `target_channels / target_environments / required_runner_os / required_runner_arches / required_runner_labels`，也可以继续把 `allowed_runner_names` 收紧到唯一 runner

### 4. 插件分发版

适用对象：

- 只想把插件接到其他 Godot 项目的团队

交付内容：

- `godot_plugin/addons/godot_agent/`

注意：

- 该形态只解决插件分发，不替代本仓库的 API、Portal、release contract 和本地验证链
- 发布前仍应从仓库主路径执行一次 `.\tools\sync_plugin.ps1`

## 支持矩阵

| 维度 | 当前状态 | 说明 |
| :--- | :--- | :--- |
| 主机 OS | `Windows` 主路径已支持 | 当前 bootstrap、plugin sync、live / browser / release 辅助脚本都以 PowerShell 为主入口 |
| Python | `3.10+` | `doctor` 明确检查 `Python 3.10 或更高版本`，`requirements.txt` 和 CLI/API 测试也围绕该版本线运行 |
| Godot | `4.x` 目标支持 | 这是基于仓库实现的推断：可执行文件探测使用 `godot / godot4` 别名，示例环境使用 `Godot_v4.4`，多处测试 fixture 使用 `4.2.0` |
| Portal browser smoke | `Chromium / Edge` | `tools/run_portal_browser_smoke.ps1` 和点击式 smoke 都按 Chrome DevTools Protocol 跑无头验证 |
| Provider profile | `codex / gemini / openai_api / claude / local_llm` | 这些 profile 已登记在 `agent_system/tools/agent_compatibility.py`，并受同一份 `agent_provider_compatibility` contract 约束 |
| Release 默认 provider 组合 | `codex + openai_api` | `release_promotion.py` 的默认 provider 选择当前固定为这两个 |
| MCP 形态 | `stdio + remote` | `bridge/tool_contracts.py` 与 `bridge/remote_mcp_server.py` 共享同一套工具 schema |
| 发布渠道 | `qa / staging / release` | `release_promotion / release_execution` 已按目标渠道输出 plan、evidence、rehearsal 与 history |

## 首次安装建议

### 1. 先预览 bootstrap 计划

```powershell
.\tools\bootstrap_clean_machine.ps1 -Preview
```

预览会返回结构化 JSON，列出：

- 将要执行的步骤
- 需要的 repo 前置文件
- 目标 report 路径
- 选中的 Python / `.venv` 路径

### 2. 再执行默认 bootstrap

```powershell
.\tools\bootstrap_clean_machine.ps1
```

默认动作：

- 创建 `.venv`
- 安装 `requirements.txt`
- 同步插件副本
- 运行 `python -m agent_system.cli doctor --report-path logs/reports/doctor_self_check.json`
- 先写出 `logs/reports/doctor_self_check.json`
- 将结果写到 `logs/reports/clean_machine_bootstrap.json`
- `clean_machine_bootstrap.json` 会额外带上 `doctor_report_path / action_item_count / blocking_checks`
- `Release Promotion Evidence / Review Bundle` 会直接复用这份报告

### 3. 需要更严格的 clean machine 验证时

```powershell
.\tools\bootstrap_clean_machine.ps1 -IncludeSmoke
```

这会在 bootstrap 后追加 targeted non-live smoke：

- `tests/test_godot_cli.py`
- `tests/test_cli.py`
- `tests/test_agent_compatibility.py`
- `tests/test_api.py`

### 4. 准备发布前再补 live/browser/remote MCP

```powershell
.\tools\run_full_live_validation.ps1
```

如果只想分段执行，可继续使用：

- `.\tools\run_live_sandbox_tests.ps1`
- `.\tools\run_portal_browser_smoke.ps1`
- `python .\tools\run_portal_browser_click_smoke.py`
- `.\tools\run_remote_mcp_live_smoke.ps1`

## 当前不承诺支持的范围

- 未经验证的 Linux / macOS 成品发布路径
- Godot 3.x
- 非 Chromium 内核的 Portal browser smoke
- 直接暴露到公网的匿名 Portal / API 部署
- 跳过 release gate、review bundle、QA evidence 的“手工约定式发布”
