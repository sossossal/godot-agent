# 制作路线图

按专业游戏制作流程看，当前项目已经具备“技术执行代理”的骨架，但还没有完全补齐“立项到发布”的制作闭环。

如果接下来要做“标准化底座”而不是继续堆单点能力，请配合阅读 [游戏开发标准化增强路线图](./游戏开发标准化增强路线图.md)。

如果当前目标已经从“技术验证”进入“准备发布成品”，请直接配合阅读 [成品发布计划](./成品发布计划.md)。

现有基础能力已经覆盖：

- `Router` 多步骤规划、执行、历史和回滚
- Godot 编辑器联动、节点注入、场景创建、资源打开
- 资源审计与低风险自动修复
- E2E 冒烟、截图、live sandbox 验证
- Portal 计划编辑器、产物中心、审计面板、编辑器状态同步
- Web 导出与基础发布产物生成

下一步不建议继续堆零散技能，而应补制作主链路。

## P0: 必须补齐

### 1. 特性验收闭环

目标:

- 让一次功能开发从“生成计划”升级到“可验收、可退回、可归档”

建议能力:

- 在计划层增加 `feature_id`、负责人、优先级、预计工时、依赖、风险
- 在步骤层增加 `验收标准`、`阻断项`、`验证方式`
- Portal 中增加 `待评审 / 待验收 / 已通过 / 已退回` 状态流转
- 执行完成后自动生成“变更摘要 + 验收清单 + 产物链接”

建议落点:

- `Task.context`
- `/plan` 与 `/execute-plan`
- Portal 计划编辑器
- `history` 与产物中心

完成标准:

- 任意一个功能都能从计划、执行、验收到归档完整追踪
- 失败任务和回滚任务能标记原因
- 能按功能维度回看“谁做了什么、什么时候通过、依赖了什么”

### 2. 数据表驱动的内容管线

状态:

- 已完成 MVP：内建 `dialogue / quest / enemy / loot / localization` 表模板，支持 schema 校验、diff 预览和导入落盘，并通过 `resource_manager` 接入主链路

目标:

- 把剧情、任务、AI 参数、战斗数值、掉落、商店、UI 文案等内容从硬编码里抽出来

建议能力:

- 定义 CSV / JSON / TSV 数据表规范与 schema 校验
- 增加导入、预览 diff、回写、导出、引用检查
- 增加对白表、任务表、敌人配置表、掉落表、本地化表模板
- 增加“修改数据表后触发相关资源检查”的自动链路

建议落点:

- `resource_manager`
- Portal 产物中心
- 审计报告与自动修复链路

完成标准:

- 策划可通过表驱动方式修改内容
- 表结构错误、字段缺失、枚举越界会被自动阻断
- 任意内容改动都能追溯到数据源和受影响资源

### 3. 发布基线与版本管理

状态:

- 已完成 MVP：发布产物现在带 `build_id / version / channel / release_summary / release_notes / release_manifest`，Web latest 会同步到稳定入口

目标:

- 把“导出 Web 项目”升级成“可交付版本管理”

建议能力:

- 每次发布生成 `build_id`、版本号、渠道、变更摘要、产物清单
- 自动输出发布说明、已知风险、回滚入口、校验信息
- 区分 `dev / qa / preview / release` 渠道
- 产物中心显示最近版本、版本间差异和回滚包

建议落点:

- `agent_system/skills/resource/export_skill.py`
- `/artifacts`
- Portal 发布区域

完成标准:

- 每个导出产物都能关联到具体任务、版本号和变更摘要
- QA 能明确知道当前测试包对应哪个功能集合
- 失败发布有标准回滚路径

## P1: 应该补齐

### 4. 断言型 QA 与性能门禁

状态:

- 已完成 MVP：发布前会按渠道执行 `feature_status` 门禁，并在可解析场景时运行 headless smoke + scene load budget，结果写入发布摘要和门禁报告

目标:

- 把现有“能跑完”的测试升级成“能判定质量是否达标”

建议能力:

- E2E 增加节点状态断言、UI 文本断言、资源存在断言
- 增加截图 diff、关键帧基线图
- 增加启动耗时、场景加载耗时、帧率、内存峰值预算
- 将性能结果写入任务产物和发布摘要

完成标准:

- 不只是测试通过，还能判断“是否达到出包标准”
- 版本回归能看到性能和画面差异

### 5. 团队协作与评审包

目标:

- 支持程序、策划、TA、QA 在同一任务视角下协作

建议能力:

- 自动汇总“改动文件、改动场景、改动资源、风险说明、验证记录”
- 生成适合 code review / design review / QA review 的摘要包
- 支持给任务附加外部链接、负责人和审批意见

完成标准:

- 一次变更无需人工整理就能进入评审
- 评审意见能回流成新步骤或阻断项

### 6. 关卡与叙事模板化生产

目标:

- 让项目能更快批量生产“可玩内容”，而不是只生成零散脚本

建议能力:

- 任务模板、敌人波次模板、关卡房间模板、对话树模板、UI 流程模板
- 将模板实例化结果直接落到场景、脚本、数据表和文档
- 给模板结果附自动验证规则

完成标准:

- 制作团队能复用模板快速产出变体内容
- 新内容生成后自动进入审计和测试链路

## P2: 可选增强

### 7. 试玩反馈与遥测回流

目标:

- 把试玩问题从口头反馈变成结构化任务输入

建议能力:

- 试玩埋点、崩溃记录、关键流程漏斗
- 玩家反馈和 QA 反馈自动转成待处理任务
- 版本维度对比关键指标

### 8. 本地化与多语言生产

目标:

- 把文本生产从单语言脚本改成多语言内容管线

建议能力:

- 字符串抽取、语言表管理、缺失翻译检查、占位符校验
- UI 文本溢出检查和字体资源检查

### 9. 版本运营与热修复准备

目标:

- 为后续 Demo、试玩节、商店页和持续更新做准备

建议能力:

- 版本标记、热修复记录、渠道差异说明
- 面向外部试玩的已知问题清单与反馈回收入口

## 推荐执行顺序

1. 先做“特性验收闭环”
2. 再做“数据表驱动内容管线”
3. 然后补“发布基线与版本管理”
4. 再把 QA 和性能门禁做成发布前置条件
5. 最后再扩团队协作、模板化生产、遥测与本地化

## 为什么这样排

- 当前项目最强的是执行层，不是管理层
- 专业制作现场真正卡进度的通常不是“生成一段代码”，而是“怎么验收、怎么批量生产内容、怎么稳定出包”
- 这三个 P0 直接决定项目能否从个人工具升级成团队生产工具
